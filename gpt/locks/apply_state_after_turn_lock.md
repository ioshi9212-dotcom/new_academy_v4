# Lock: после хода обязательно применять изменения в state

Этот файл имеет высокий приоритет для каждого игрового хода.

## Главная проблема

Текст сцены не является сохранённым состоянием мира.

`current_state.json` тоже не заменяет остальные state-файлы.

Если после сцены изменились события, отношения, знания, слухи, репутация, долги, обещания, инвентарь, дата, текущая сцена, открытые нитки или состояние персонажей, эти изменения нужно применить в соответствующие файлы `state/*.json`.

Нельзя считать изменения сохранёнными, если они остались только:

- в тексте сцены;
- в ответе GPT;
- в `current_state.json`;
- в `scene_text`;
- в `turn_results` без применения;
- в памяти чата.

Если сцена изменила события/отношения/репутацию/знания/долги, но нужный state-файл не обновлён, ход считается технически незавершённым.

## Главный рабочий файл событий

Для событий и сюжетных последствий использовать:

```txt
state/story_lines.json
```

Именно туда записывать:

- события по датам;
- `shared_events`;
- линии Хару/Райдена/Ливии/Киары/Академии/репутации/слухов/обязательств;
- долги, ставки, обещания;
- прогресс повторяющихся ниток;
- кто видел событие;
- кто слышал событие;
- кому рассказали;
- кто что узнал;
- кто НЕ знает;
- источник знания;
- какие линии получили эффект;
- что нужно сжать каждые 15 ходов.

Нельзя создавать новый одноразовый `state/*.json` под каждую сцену как основную память.

## Knowledge Source Gate

Перед применением state после сцены ИИ обязан разделять:

```txt
память автора / факт мира
память конкретного персонажа
```

`state/story_lines.json` и `state/scene_history.json` — это память автора.

`state/knowledge_state.json` и поля `known_by/told_to/heard_by/public_to` — это память персонажей.

Нельзя добавлять знание персонажу и нельзя давать персонажу будущие реплики на основе события, если нет источника знания.

Источник знания может быть только:

- `participants` — персонаж участвовал;
- `witnesses` — персонаж видел;
- `heard_by` — персонаж слышал;
- `told_to` — персонажу рассказали;
- `public_to` — событие стало публичным для группы;
- `known_by` — персонаж явно знает;
- `knowledge_state.characters.<id>.knows` — факт уже записан;
- `rumors_state` — слух реально дошёл до персонажа/группы;
- должностной доступ — только если он указан в карточке или `knowledge_state`.

Если источника нет, персонаж не знает факт.

Он может подозревать, спрашивать, ошибаться или реагировать на видимое поведение, но не утверждать.

## Обязательная структура события после сцены

Каждое новое важное событие в `story_lines_changes` / `state/story_lines.json.shared_events` должно иметь:

```json
{
  "event_id": "short_stable_id_with_date",
  "date": "1198-08-17",
  "time": "утро",
  "location": "столовая",
  "participants": [],
  "witnesses": [],
  "heard_by": [],
  "told_to": [],
  "public_to": [],
  "known_by": [],
  "not_known_by": [],
  "source_notes": {},
  "summary": "Коротко что произошло.",
  "line_refs": []
}
```

Если `known_by` не указан, известность события считается равной `participants + witnesses + heard_by + told_to + public_to`.

Если ни одно поле доступа не заполнено, событие считается известным только `participants`.

## Обязательная структура scene_history

`state/scene_history.json` после сцены должен хранить краткий итог с доступом к знанию:

```json
{
  "scene_id": "scene_id_with_date",
  "date": "1198-08-17",
  "time": "утро",
  "location": "столовая",
  "participants": [],
  "witnesses": [],
  "heard_by": [],
  "told_to": [],
  "public_to": [],
  "known_by": [],
  "not_known_by": [],
  "summary": "Краткий технический итог сцены.",
  "state_changes": []
}
```

`scene_history.summary` не становится знанием всех NPC.

## Обязательная цепочка после каждой сцены

После написания сцены ИИ обязан:

1. Определить, что реально изменилось.
2. Разложить изменения по конкретным state-файлам.
3. Если есть событие/долг/обещание/слух/линию — подготовить `story_lines_changes`.
4. В каждом событии указать `participants/witnesses/heard_by/told_to/public_to/known_by/not_known_by/source_notes`.
5. Если есть изменение отношений — подготовить `relationship_changes` и/или краткий эффект в `story_lines`.
6. Если кто-то что-то узнал — подготовить `knowledge_changes` с источником знания.
7. Если персонаж не присутствовал и нет `told_to/public_to/rumor`, не добавлять ему знание.
8. Если изменилась репутация/слухи — подготовить `reputation_changes`/`rumors_changes` с областью распространения.
9. Применить изменения через API:
   - предпочтительно `/api/v1/sessions/{session_id}/apply-turn-result`, если доступен;
   - или прямым `PUT /api/v1/sessions/{session_id}/json/{file_path}`, если нужно точечно обновить файл.
10. Убедиться, что изменились именно нужные файлы.
11. Только после этого считать ход завершённым.

Если изменений нет — честно указать, что state не менялся.

Если инструмент записи недоступен — не делать вид, что состояние сохранено. Нужно прямо сообщить: `state не обновлён, потому что запись недоступна`.

## Даты обязательны

Каждое событие, нитка, долг, progress-log и запись в timeline должны иметь абсолютную дату.

Формат:

```txt
1198-08-17
```

Перед использованием слов `сегодня`, `вчера`, `позавчера`, `несколько дней назад`, `неделю назад` сверять дату события с `state/current_state.json` и `state/story_lines.json`.

Нельзя говорить `вчера`, если событие было больше одного дня назад.

## Не сохранять только current_state

`state/current_state.json` хранит текущую сцену, место, время, active/nearby персонажей и краткий якорь.

Он НЕ должен быть единственным местом, куда попали последствия сцены.

Если в сцене были:

- личная реакция;
- рост интереса;
- уважение;
- ревность;
- конфликт;
- обида;
- флирт;
- обещание;
- договорённость;
- долг;
- ставка;
- социальное внимание;
- слух;
- новое знание;
- изменение репутации;
- заметка инструктора;
- будущий крючок;

то нужно обновить профильный state-файл и/или `state/story_lines.json`, а не только `current_state.json`.

## Отношения

Если изменилась личная динамика между персонажами, обновить:

```txt
state/relationships.json
```

и, если событие важно для линии, также добавить короткий эффект в `state/story_lines.json`.

Пары писать в формате:

```txt
character_a__character_b
```

Для важных пар нельзя сохранять только цифры. Нужны хотя бы `status`, `notes`/`memory`, `open_threads` или `last_interaction`, если сцена значимая.

## Knowledge state

Если персонаж узнал новый факт, обновить:

```txt
state/knowledge_state.json
```

Фиксировать:

- сам факт;
- источник знания;
- кто видел;
- кто слышал;
- кому рассказали;
- кто сделал вывод;
- кто ошибочно понял;
- кто не знает;
- чьё имя Акира уже знает;
- кого Акира видела, но имени не знает.

Hidden lore нельзя добавлять в `knows`, если персонаж не узнал это сценой.

Нельзя добавлять знание персонажу только потому, что событие есть в `story_lines` или `scene_history`.

## Reputation / Rumors

Если изменилось отношение группы/академии к персонажу, обновить:

```txt
state/reputation_state.json
```

Если появился слух, обновить:

```txt
state/rumors_state.json
```

Если слух/репутация связаны с событием, также записать event/effect в `state/story_lines.json` в линии `line_reputation` или `line_social_media_rumors`.

Слух должен иметь:

- источник;
- область распространения;
- кто мог видеть;
- кто точно знает;
- кто только слышал и может ошибаться;
- уровень правды;
- дату появления.

Слухи не должны мгновенно знать все.

## Obligations / ставки / долги

Ставки, обещания и долги хранить не отдельными файлами, а в:

```txt
state/story_lines.json -> line_obligations
```

Когда долг двигается, обновлять ту же запись.

## Current state

Если изменились дата, время, место, active/nearby персонажи, состояние Акиры или цель сцены, обновить:

```txt
state/current_state.json
```

Особенно важно обновлять:

- `current_location_id`;
- `date`;
- `time_of_day`;
- `active_characters`;
- `nearby_characters`;
- `speaking_character_ids`;
- `observing_character_ids`;
- `addressed_character_ids`;
- `looked_at_character_ids`;
- `current_scene_goal`;
- `visible_inventory`;
- `nearby_items`;
- `last_scene_anchor`.

## Scene history и open threads

Если в сцене было что-то, что должно вернуться позже, обновить:

- `state/story_lines.json` как основной файл линий;
- `state/scene_history.json` как краткую историю;
- `state/current_state.json.open_threads`, если нитка нужна прямо в ближайших ходах.

Нельзя держать договорённость только в тексте сцены.

## Что запрещено

Запрещено:

- создавать одноразовые `state/*.json` под каждую сцену;
- писать “отношения изменились”, но не обновлять `relationships.json` или `story_lines`;
- обновлять только `current_state.json`, если изменились отношения/репутация/знания/события;
- оставлять изменения только в `turn_results`;
- сохранять крупные эмоциональные сдвиги без конкретной сцены;
- добавлять знание персонажу, если он не мог его получить;
- добавлять знание персонажу без источника;
- считать `scene_history.summary` знанием всех персонажей;
- считать `story_lines.current_summary` знанием всех персонажей;
- удалять старые notes/memory/open_threads без причины;
- перезаписывать весь файл пустым/урезанным состоянием;
- менять future locks как уже случившийся факт;
- писать в отчёте “состояние обновлено”, если API-запись не была выполнена;
- использовать `вчера` без сверки абсолютной даты.

## Проверка перед финальным ответом

Перед финальным ответом после сцены проверить:

- были ли реальные изменения событий/линий;
- нужно ли обновить `state/story_lines.json`;
- есть ли в каждом новом event `known_by/not_known_by/source_notes`;
- были ли реальные изменения отношений;
- были ли изменения репутации;
- были ли новые знания персонажей;
- у каждого нового знания есть ли источник;
- были ли слухи;
- появились ли открытые нитки;
- какие state-файлы должны быть обновлены;
- были ли они сохранены через API.

Если событие/отношения/репутация/знания изменились, но соответствующий state-файл не обновлён — ход считается незавершённым.

Если запись недоступна, нужно прямо сказать об этом, а не продолжать вид, что память сохранена.
